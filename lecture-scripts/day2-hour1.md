今天開始講容器技術。

這叫「It works on my machine」——業界經典的痛。

今天就是要解決這個問題。

---

## 二、環境一致性問題（10 分鐘）

### 問題根源

程式不是獨立存在的，它依賴：
- 作業系統版本（Ubuntu 20.04 vs 22.04）
- 程式語言版本（Python 3.8 vs 3.11）
- 套件版本（Django 3.2 vs 4.2）
- 系統設定（環境變數、設定檔）

這些加起來叫「執行環境」。

開發機、測試機、正式機——三個環境通常都不一樣。

### 真實案例

你用 Mac + Python 3.11 + Django 4.2 開發。
丟到公司 Linux 伺服器，Python 是 3.8。
Django 4.2 有些語法 3.8 不支援，直接報錯。

維運說：「你程式有問題」
你說：「在我電腦明明可以跑」

經典對話。

### 傳統解法

**1. 寫詳細安裝文件**
→ 會過時、會漏寫、不同人照做結果不同

**2. 用虛擬機**
→ 可以解決，但太肥。一個 VM 幾十 GB，開機要幾分鐘，一台機器跑不了幾個。

## 三、容器是什麼（15 分鐘）

### 核心概念

容器 = 輕量級虛擬化

不模擬整個作業系統，只打包應用程式 + 相依套件。

靠 Linux 核心兩個功能實現：
- **Namespace**：隔離。每個容器有獨立的程序空間、網路、檔案系統。
- **Cgroups**：限制資源。控制 CPU、記憶體用量。

容器共用主機的 Linux 核心，所以很輕。

### 容器 vs 虛擬機

```
虛擬機：
┌─────────┐ ┌─────────┐
│  App A  │ │  App B  │
├─────────┤ ├─────────┤
│ Guest OS│ │ Guest OS│  ← 每個都裝完整 OS
└─────────┘ └─────────┘
┌─────────────────────┐
│     Hypervisor      │
├─────────────────────┤
│      Host OS        │
└─────────────────────┘

容器：
┌─────────┐ ┌─────────┐
│  App A  │ │  App B  │
├─────────┤ ├─────────┤
│  Libs   │ │  Libs   │
└─────────┘ └─────────┘
┌─────────────────────┐
│   Container Engine  │  ← 共用核心
├─────────────────────┤
│      Host OS        │
└─────────────────────┘
```

### 比較表

| 項目 | 虛擬機 | 容器 |
|-----|-------|------|
| 啟動時間 | 分鐘 | 秒 |
| 大小 | GB | MB |
| 效能 | 有損耗 | 接近原生 |
| 單機數量 | ~10 | 數百 |

## 四、Docker 簡介（15 分鐘）

### Docker 是什麼

2013 年開源，目前最流行的容器平台。

把容器技術變得簡單好用。

一句話：**把應用程式和執行環境打包成標準單元，在任何有 Docker 的機器上都能跑。**

### 核心三元素

**1. Image（映像檔）**
- 唯讀模板
- 包含程式 + 環境 + 設定
- 像光碟

**2. Container（容器）**
- Image 的執行實例
- 可以啟動、停止、刪除
- 像用光碟裝好的電腦

**3. Registry（倉庫）**
- 存放 Image 的地方
- Docker Hub = 最大公開倉庫
- 像 App Store

### 如何解決環境問題

1. 開發者建立 Image（程式 + 環境都打包）
2. 推送到 Registry
3. 維運拉下來直接跑

環境一致，問題解決。

不只給程式碼，連環境一起給。
