## 為什麼需要 Kubernetes

隨著容器化技術的快速發展，越來越多的企業和開發者都開始使用微服務和容器化技術來構建自己的服務和應用程式。容器化的優點非常多，比如可以更加快速和高效地開發和部署應用程式，還可以提高應用程式的可用性和穩定性，最重要的也幫助企業提高了資源的利用效率、降低了成本。

除了上面提到的這些優點之外，當然也帶來了一些新的問題。這裡面最突出的一個問題就是：容器的數量會變得非常多，架構也越來越複雜。那麼如何管理和維護這些容器，就成了一個非常大的問題和挑戰。容器數量少的話，可以使用一些簡單腳本來管理，但是當容器數量達到一定規模，比如成百上千、甚至數以萬計的時候，使用腳本來管理就會變得非常困難，甚至是不可能的。

而 Kubernetes 就是為了解決這個問題而生的，它可以幫助我們優雅地管理容器化的應用程式和服務。

Kubernetes 是由 Google 開源的容器編排引擎，它可以幫助我們管理容器化的應用程式和服務，也是目前最流行的容器編排引擎，主要用於容器化應用程式的部署、擴展和管理等等。

Kubernetes 是如何解決這些問題的呢？它又提供了哪些特性呢？

首先一點就是它提供了**容器編排**的功能，可以通過一些配置檔案來定義應用程式的部署方式，讓容器的建立、維護和管理變得更加簡單和高效。

**高可用**是指系統可以在長時間內持續正常地運行，並不會因為某一個組件或者服務的故障而導致整個系統不可用這樣的情況發生。Kubernetes 也提供了很多高可用的特性，比如自動重啟、自動重建、自我修復等等。這些特性可以幫助我們提高叢集的可用性，從而讓使用者可以在任何時間內都能正常地使用系統。

**可擴展性**是指系統可以根據負載的變化來動態地擴展或者縮減系統的資源，從而來提高系統的效能和資源的利用率。最典型的例子就是每年雙十一或者 618 的時候，各大電商平台和網路企業的流量都會暴增。這個時候比較好的一個解決方案就是根據負載的變化來動態地擴展或者縮減系統的資源，比如縮減一些不太重要的服務或者測試環境，然後再增加一些關鍵的服務資源，這樣就可以保證系統平穩地度過流量高峰時段，提高使用者的體驗。

還有很多其他特性，比如災難恢復、彈性伸縮等等，時間關係這裡就不一一列舉了。這些特性可以幫助我們提高應用程式的效能、可用性和穩定性。另外良好的生態系統和社群支持，也為 Kubernetes 的發展提供了很大的動力。可以說目前 Kubernetes 已經成為了容器編排領域的行業標準。

## K8s 核心概念與資源物件

這節課我們來了解一下 Kubernetes 中的一些常用的核心概念以及它們之間的關係。到目前為止 Kubernetes 中大大小小的資源物件已經有不下幾十種了，但是大多數時間我們其實只需要使用其中幾個最為核心的資源物件就可以了。

下面就從一個最簡單的例子開始，通過架構逐漸升級和演進的過程，來一步步引入 Kubernetes 中的各種資源物件。這樣不但可以理解它們是為了解決什麼問題而存在的，還可以理解它們之間是如何協同工作的。這樣大家就可以根據需求來選擇使用哪些資源物件，以及如何使用這些資源物件來解決自己的問題了。

### Node 與 Pod

首先從一個最簡單的節點開始。一個節點就是一台實體機或者虛擬機，在這個節點上我們可以運行一個或者多個 Pod。Pod 是 Kubernetes 最小調度單元，一個 Pod 就是一個或者多個應用容器的組合。它建立了一個容器的運行環境，在這個環境中容器可以共享一些資源，比如網路、儲存以及一些執行時的配置等等。

假設我們的系統包括一個應用程式和一個資料庫，就可以將應用程式和資料庫分別放到兩個不同的 Pod 中。一般情況下我們建議一個 Pod 中只運行一個容器，但是並不是說一個 Pod 中只能運行一個容器，這更多的是一種最佳實踐。因為這樣可以更好地實現應用程式的解耦和擴展。

其實一個 Pod 中是可以運行多個容器的，但是一般也僅限於這些容器是高度耦合的情況，它們之間為了實現某種功能或者共享某些資源而不得不放到同一個 Pod 中。比如大家經常聽到的 Sidecar，也就是邊車模式，就是將一個應用容器和一個輔助容器放到同一個 Pod 中。這個輔助容器就是 Sidecar，通常用來實現一些輔助的功能，比如日誌收集、監控或者配置管理等等。

### Service

現在我們已經將應用程式和資料庫分別放到兩個 Pod 中。應用程式要存取資料庫的話，就需要知道資料庫的 IP 位址。這個 IP 位址是在 Pod 建立的時候自動分配的，它是一個叢集內部的 IP 位址，Pod 之間可以通過這個 IP 位址來進行叢集通訊。這樣我們的應用程式就可以通過這個 IP 位址來存取資料庫了。

但是這裡有兩個問題需要注意一下。一個是這個 IP 位址是一個內部的 IP 位址，在叢集外部是無法存取的。還有一個問題就是 Pod 並不是一個穩定的實體，也就意味著它們非常容易被建立或者銷毀。比如發生故障的時候，Kubernetes 會自動將 Pod 銷毀掉，再建立一個新的 Pod 來替代它。這個時候 Pod 的 IP 位址也會發生變化，如果應用程式還是用原來的 IP 位址來存取資料庫的話，就會出現問題了。

為了解決這個問題，Kubernetes 提供了一個叫做 **Service** 的資源物件。Service 翻譯過來也就是服務，它可以將一組 Pod 封裝成一個服務，這個服務可以通過一個統一的入口來存取。拿剛剛的例子來說，我們可以將應用程式和資料庫的 Pod 分別封裝成兩個 Service。這樣的話，應用程式就可以通過 Service 的 IP 位址來存取資料庫了。這樣當 Pod 發生故障的時候，即使它的 IP 位址發生了變化，但是 Service 的 IP 位址是不會發生變化的。Service 會自動將請求轉發到其他健康的 Pod 上，這樣就解決了 Pod 的 IP 位址不穩定的問題。

服務也分為內部服務和外部服務。內部服務是一些我們不想暴露給外部或者沒有必要暴露給外部的服務，比如圖中的 MongoDB 資料庫就是一個例子，其他類似的還有快取、訊息佇列等等。這些服務只需要在叢集內部被存取就可以了，這種類型的服務我們稱之為內部服務。

相應地，有些服務是需要暴露給外部的，比如一些微服務的後端 API 介面，或者是一些給使用者使用的前端介面等等。這種類型的服務我們稱之為外部服務。外部服務有幾種常用的類型，其中一種類型就是 **NodePort**，它會在節點上開放一個 Port，然後將這個 Port 映射到 Service 的 IP 位址和 Port 上。這樣我們就可以通過節點的 IP 位址和 Port 來存取 Service 了。

### Ingress

在開發或者測試階段，使用這種 IP 和 Port 的方式是沒有問題的，但是在生產環境中通常都是通過域名來存取服務的。這個時候就用到了另一個資源物件 — **Ingress**。Ingress 是用來管理從叢集外部存取叢集內部服務的入口和方式的，可以通過 Ingress 來配置不同的轉發規則，從而根據不同的規則來存取叢集內部不同的 Service，以及 Service 所對應的後端 Pod。還可以通過 Ingress 來配置域名，這樣就可以將原本使用 IP 位址和 Port 的方式轉換成使用域名的方式來存取 Service 了。另外 Ingress 也可以配置一些其他的功能，比如負載均衡、SSL 憑證等等，這些功能我們會在後面的課程中詳細介紹。

### ConfigMap 與 Secret

現在我們的應用程式和資料庫已經可以在 Kubernetes 中運行並且對外提供服務了，但是這個時候我們還沒有考慮到一個問題，就是應用程式和資料庫之間的耦合問題。比如我們的應用程式需要存取資料庫的話，通常的做法是把資料庫的位址和 Port 等連線資訊寫到配置檔或者環境變數中，然後在應用程式中讀取這些配置資訊。但是這樣的話，配置資訊就和應用程式耦合在一起了。一旦資料庫的位址或者 Port 發生了變化，那麼我們就需要重新編譯應用程式，然後再重新部署到叢集中。這樣不但非常麻煩，而且還會造成應用程式的停機時間。這對於一些需要 7x24 小時不間斷運行的系統或者服務來說，顯然是不能接受的。

為了解決這個問題，Kubernetes 提供了一個叫做 **ConfigMap** 的組件。它可以將一些配置資訊封裝起來，然後就可以在應用程式中讀取和使用了。有了 ConfigMap，我們就可以把配置資訊和應用程式的映像檔內容分離開來，這樣就可以保持容器化應用程式的可移植性。當資料庫的位址和 Port 發生變化的時候，我們也只需要修改 ConfigMap 物件中的配置資訊，然後重新載入 Pod 就可以了，不需要重新編譯和部署應用程式。這樣就實現了應用程式和資料庫的解耦。

有一點需要注意的是，ConfigMap 中的配置資訊都是明文的。如果配置資訊中包含一些敏感資訊，比如資料庫的使用者名稱或者密碼等等，那麼這些敏感資訊就不建議儲存在 ConfigMap 中，因為這樣會帶來一些安全風險和問題。

為了解決這個問題，Kubernetes 提供了另一個叫做 **Secret** 的組件。和 ConfigMap 類似，它可以將一些敏感資訊封裝起來，然後就可以在應用程式中讀取和使用了。需要注意的是，這些敏感資訊雖然不是以明文的形式來儲存的，但是也只是做了一層 Base64 的編碼而已。Base64 編碼只是一種編碼方式，並不是一種加密方式，所以僅僅只靠 Secret 並不能完全保證敏感資訊的安全，還需要配合一些其他的手段來提高安全性，比如 Kubernetes 提供的一些其他安全機制，包括網路安全、存取控制、身份認證等等。

### Volume

當容器被銷毀或者重啟的時候，容器中的資料也會跟著消失。這對於一些需要持久化儲存的應用程式，比如資料庫來說顯然是不行的。為了解決這個問題，Kubernetes 提供了一個叫做 **Volume** 的組件。它可以將一些持久化儲存的資源掛載到叢集中的本地磁碟上，或者掛載到叢集外部的遠端儲存上。這樣即使容器被銷毀或者重啟，這些資料也不會遺失，也就實現了容器中資料的持久化儲存。

### Deployment 與 StatefulSet

到目前為止，我們的程式已經可以在 Kubernetes 中運行並且對外提供服務了，但是這個時候我們還沒有考慮到一個問題，那就是應用程式的高可用性。比如如果應用程式所在的節點發生了故障，或者需要對節點進行升級和更新維護的時候，應用程式就會停止服務，這樣就有可能造成一些不必要的麻煩和損失。

解決方案其實也非常簡單：既然只有一個節點不行的話，那麼就多加幾個節點。簡單來說就是把所有東西都複製一份，然後放到另外一個節點上。這樣當一個節點發生故障的時候，Service 會自動將請求轉發到另一個節點上來繼續提供服務。

Kubernetes 提供了一個叫做 **Deployment** 的組件就是用來解決這個問題的。可以定義和管理應用程式的副本數量以及應用程式的更新策略，可以簡化應用程式的部署和更新操作。在之前我們提到過，Pod 可以理解為在容器的上面加了一層抽象，這樣就可以將一個或者多個容器組合在一起。而 Deployment 就可以理解為在 Pod 的上面再加上一層抽象，這樣就可以將一個或者多個 Pod 組合在一起，並且還具有副本控制、滾動更新、自動擴縮容等高級特性和功能。

**副本控制**就是指可以定義和管理應用程式的副本數量。比如我們可以定義一個應用程式的副本數量為 3 個，那麼如果有副本發生故障的時候，它會自動建立一個新的副本來替代它，始終保持有 3 個副本在叢集中運行。

**滾動更新**就是指可以定義和管理應用程式的更新策略，可以輕鬆地升級應用程式的版本，逐漸使用新的版本替換掉舊的版本，確保應用程式的平滑升級。

除了應用程式之外，資料庫也會存在類似的問題。比如資料庫的節點發生了故障，或者需要升級和更新維護的時候，也會停止服務影響使用者的正常使用。所以資料庫一般也是需要採取類似的多副本的方式來保證它的高可用性的。但是和應用程式不同的是，我們一般不使用 Deployment 來實現資料庫的多副本，因為資料庫的多副本之間是有狀態的。簡單來說就是每個副本都有自己的狀態，也就是資料。這些資料都是需要持久化儲存的，而且也需要保證多個副本之間的資料是一致的，這就需要一些額外的機制，比如把資料寫入到一個共享的儲存中，或者把資料同步到其他的副本中。

對於這一類有狀態的應用程式，Kubernetes 提供了另一個叫做 **StatefulSet** 的組件來管理。StatefulSet 和 Deployment 非常類似，也提供了定義和管理應用程式副本數量，或者動態擴縮容等等功能。此外它還保證了每個副本都有自己穩定的網路標識、服務和持久化儲存。因此像資料庫、快取、訊息佇列等等這些有狀態類的應用，以及一些保留了會話狀態的應用程式，一般都需要使用 StatefulSet 而不是 Deployment 來部署。

StatefulSet 並不是萬能的，部署的過程還是比較複雜和繁瑣的，而且並不是所有的有狀態應用都適合使用 StatefulSet 來部署。一種更加通用和簡單的方式是把資料庫這種有狀態的應用程式從 Kubernetes 叢集中剝離出來，在叢集外單獨部署。這樣可以避免很多不必要的麻煩和問題，而且可以簡化叢集的架構和管理。

現在我們的應用程式和資料庫都已經有了多個副本，這樣即使某個節點完全發生故障，比如發生了硬體故障或者系統崩潰等等，其他節點也可以繼續對外提供服務，並不會影響到使用者的正常使用，直到我們修復好這個節點然後將它重新加入到叢集中。這樣就避免了單點故障的問題，減少了系統發生故障停機的時間，提高了系統的可用性。

### 核心概念小結

來總結一下本節課的內容：

- **Node**：它是 Kubernetes 叢集中的一個節點，可以是實體機也可以是虛擬機。
- **Pod**：它是 Kubernetes 中的最小調度單元，也可以理解成是容器的一個抽象，可以是一個或者多個容器的組合，但是一般來說一個 Pod 中只運行一個容器。
- **Service**：可以將一組 Pod 封裝成一個服務，並且提供一個統一的入口來存取這個服務。然後為了對外提供服務，我們還引入了 **Ingress**，將請求路由轉發到叢集內部的 Service 上。
- **ConfigMap 和 Secret**：它們可以將一些配置資訊和敏感資訊封裝起來，供應用程式使用，避免了配置變更帶來的重新編譯和部署的問題。
- **Volume**：它可以將一些資料掛載到叢集中的本地磁碟或者遠端儲存上，從而實現了叢集中資料的持久化儲存。
- **Deployment 和 StatefulSet**：分別用來部署無狀態和有狀態的應用程式。它們可以將一個或者多個 Pod 組合在一起，並且具有副本控制、滾動更新、自動擴縮容等高級特性和功能，從而實現應用程式的高可用性。

## K8s 架構與核心組件

這節課我們來看一下 Kubernetes 的架構，包括它的一些核心組件以及它們之間的關係和工作原理。

Kubernetes 是一個典型的 Master-Worker 架構。Master Node 負責管理整個叢集，Worker Node 負責運行應用程式和服務。簡單來說就像是一個公司的管理層和員工一樣，管理層負責整個公司的管理和運營，員工負責完成公司的實際工作。

官方給出的定義是：Kubernetes 會將容器放入在節點上運行的 Pod 中，來執行你的工作負載。拿上節課的例子來說的話，在一個 Node 上運行了兩個 Pod，分別是應用程式和資料庫。這裡的 Node 就是 Kubernetes 叢集中真正完成實際工作的節點，是真正幹活的機器，這也是為什麼有的時候它會被叫做工作節點（Worker Node）。

### Worker Node 組件

為了能夠對外提供服務，每個 Node 上都會包含三個組件，分別是 **kubelet**、**kube-proxy** 和 **Container Runtime**（容器執行時期）。

**Container Runtime（容器執行時期）**的概念稍微有些抽象，你可以把它理解成一個運行容器的軟體。它負責拉取容器映像檔、建立容器、啟動或者停止容器等等。所有的應用程式都需要使用容器執行時期來運行，所以每個工作節點上都必須要安裝容器執行時期。在學習 Docker 的時候，我們已經接觸過了一個容器執行時期，就是 Docker Engine。而在 Kubernetes 中，除了 Docker Engine 之外，你可以使用任何你喜歡的容器執行時期，常見的容器執行時期包括 containerd、CRI-O 等。

工作節點上另一個主要的服務是 **kubelet**。它負責管理和維護每個節點上的 Pod，並確保它們按照預期運行。它也會定期從 API Server 組件接收新的或者修改後的 Pod 規範，同時它也會監控工作節點的運行情況，然後將這些資訊回報給 API Server。

**kube-proxy** 是工作節點的另一個核心組件，負責為 Pod 物件提供網路代理和負載均衡服務。通常情況下，Kubernetes 叢集可能包含多個節點，這些節點之間通過 Service 來進行通訊，這就需要一個負載均衡器來接收請求，然後再將請求傳送到不同的節點上來完成負載均衡的功能。kube-proxy 就是負責這個功能的組件，它會在每個 Node 上啟動一個網路代理，使發往 Service 的流量以一種高效的方式路由到正確的後端 Pod 中。比如我們的應用程式要存取資料庫的時候，它的存取並不會被隨機路由到任意一個節點上，而會被路由到和應用程式在同一個節點上的那個資料庫的 Pod 中，節省了把請求轉發到其他節點上的網路開銷。

### Master Node 組件

以上就是工作節點上的三個組件以及它們之間的關係和工作原理。那麼叢集中是如何管理這些節點的呢？比如如何調度不同的 Pod 到不同的節點上、如何監控節點的狀態、如何新增或者刪除一個節點？這些就是 Master Node 要解決的問題了。

Master 節點上的組件和 Worker 節點上的組件完全不同，Master 節點上有四個基本組件，分別是 **kube-apiserver**、**etcd**、**Controller Manager** 和 **Scheduler**。

#### kube-apiserver

第一個組件是 **kube-apiserver**。它負責提供 Kubernetes 叢集的 API 介面服務，所有的組件都會通過這個介面來進行通訊。比如當你作為一個使用者，需要在叢集中部署一個新的應用的時候，就需要通過一些客戶端來與 API Server 進行互動。這個客戶端可以是 kubectl 命令列工具，也可以是一些其他的客戶端，比如 Dashboard 或者其他的一些圖形化 UI 工具。

API Server 就像是一個叢集的閘道器，是整個系統的入口。所有的請求都會先經過它，然後再由它來分發給不同的組件進行處理。比如一些建立、更新、刪除 Pod 的請求，或者是一些查詢叢集狀態的命令，都會先經過 API Server，再轉發給相應的組件來處理。

除了提供 API 介面服務之外，API Server 還負責對所有資源物件的增刪改查等操作進行認證、授權和存取控制，確保只有經過認證和授權的使用者才能夠存取到叢集中的資源物件，這樣就提高了整個叢集系統的安全性。

#### Scheduler

舉個例子，當我們使用 kubectl 命令列工具來建立一個新的 Pod 的時候，這個請求會先到達 API Server，然後 API Server 會驗證這個請求的合法性，比如你是否有權限來建立這個 Pod。驗證通過之後，才會繼續將請求轉發給相應的組件來處理。這個組件也就是我們接下來要介紹的第二個組件 — **Scheduler**（調度器）。

它負責監控叢集中所有節點的資源使用情況，然後根據一些調度策略將 Pod 調度到合適的節點上運行。比如當我們新增一個 Pod 的時候，如果第一個節點上的資源占用了 80%，而第二個節點上的資源只占用了 20%，那麼這個 Pod 就會被智慧地調度到第二個節點上，這樣就可以更加合理地利用叢集中的各種資源。

#### Controller Manager

下一個組件是 **Controller Manager**（控制器管理器）。它負責管理叢集中各種資源物件的狀態。比如當叢集中任何一個節點上的 Pod 發生故障的時候，必須要有一種機制來監控和偵測到這個故障，然後盡可能快地對這個故障進行處理，比如重新啟動這個 Pod 或者使用其他的 Pod 來代替它。這個就是 Controller Manager 要做的事情，它會監控叢集中各種資源物件的狀態，比如 Pod、Node、Service 等等，然後根據這些狀態來做出相應的回應，確保叢集中的各種資源物件都處於我們預期的狀態。

#### etcd

那麼它是如何知道叢集中各種資源物件的狀態是否正常的呢？這就涉及到了 Master 節點中的最後一個組件 — **etcd**。它是一個高可用的鍵值儲存系統。看過我們之前 Redis 課程的同學應該對這個概念比較熟悉。etcd 和 Redis 比較類似，也是一個鍵值儲存系統，用來儲存叢集中所有資源物件的狀態資訊。比如某一個 Pod 掛掉崩潰了，或者增加了一個新的 Pod 等等，這些資訊都會被記錄到 etcd 中。可以把它理解為一個叢集的大腦，是整個叢集的資料儲存中心。很多功能都是依賴它來實現的，比如我們使用命令列或者 UI 介面來查詢整個叢集的狀態，這些資訊都是從 etcd 中取得的。

這裡需要注意的一點是，etcd 中一般只儲存叢集中各種資源物件的狀態資訊，而應用程式中的資料，比如我們資料庫中的資料，一般是不會儲存在 etcd 中的。

#### Cloud Controller Manager

如果你使用的是雲服務商，比如 Google 或者 AWS 提供的 Kubernetes 叢集，那麼還會有一個額外的組件叫做 **Cloud Controller Manager**（雲控制器管理器）。它是一個雲平台相關的控制器，負責與雲平台的 API 進行互動，並且可以提供一致的管理介面，使得使用者可以更加方便地在不同的雲平台中運行和管理它們的應用程式。

## 環境搭建：Minikube

下面來看一下什麼是 Minikube 和 kubectl，以及如何在本地搭建一個 Kubernetes 環境。

**Minikube** 是一個輕量級的 Kubernetes 發行版，它可以在本地運行一個單節點的 Kubernetes 叢集。在生產環境的 Kubernetes 叢集一般都是由多個節點組成的，每個節點都是一台伺服器或者虛擬機。比如兩台 Master 節點和三個 Worker 節點的話，就需要五台伺服器或者虛擬機，這樣的叢集才能保證高可用性。

但是在本地開發環境中，搭建一個這樣的多節點 Kubernetes 叢集是非常麻煩的，需要多台伺服器或者虛擬機，而且還需要配置一些網路和儲存等環境，這樣的環境搭建起來非常複雜，有的時候受限於硬體資源甚至是不可行的。所以在本地開發環境中，我們就可以使用 Minikube 來搭建一個簡化版的 Kubernetes 叢集，來模擬生產環境進行一些開發測試或者部署驗證。

Minikube 是一個能夠讓我們在本地運行 Kubernetes 的工具。它能夠在本地運行一個單節點的 Kubernetes 叢集，這個叢集是在你電腦中的一個虛擬機裡面運行的，非常適合用來學習或者本地開發測試一些 Kubernetes 應用。

除了 Minikube 之外，其他一些工具比如 K3s、K3d、Kind 等也可以用來搭建本地的輕量級 Kubernetes 環境。這節課我們先來了解一下 Minikube，下節課再來看一下 K3s 的使用，其他環境感興趣的同學可以自行了解一下。

### kubectl 簡介

除了搭建一個 Kubernetes 叢集環境之外，你還需要使用 **kubectl** 這個命令列工具來同你建立的 Kubernetes 叢集進行互動，比如建立一個 Pod 或者 Service、查看叢集的狀態等等。

在上節課介紹 Kubernetes 架構的時候，我們提到過在 Kubernetes 叢集中有一個 Master 節點，這個 Master 節點上有一個叫做 API Server 的組件，它是 Kubernetes 叢集的 API 介面。我們可以通過這個介面來管理叢集中的資源。而同 API Server 進行互動的方式包括：使用 Dashboard 這樣的 Web UI 介面、也可以使用程式封裝 API 介面的方式、或者使用命令列工具。kubectl 就是這樣一種命令列工具，可以通過在命令列輸入各種命令的方式來同 Kubernetes 叢集進行互動，管理叢集中的各種資源。

kubectl 也是這三種方式中最為常用和功能最強大的一種，基本上可以執行你想要的任何操作。而且不止我們本地的 Minikube 叢集，你也可以和任何其他的 Kubernetes 叢集進行互動，比如你的生產環境叢集，或者部署在雲端的叢集。只要你能夠存取到這個叢集的 API 介面並且有足夠的權限，就可以使用 kubectl 來管理這個叢集。我們的課程中也會主要使用 kubectl 來進行操作和演示。

簡單總結一下：Minikube 用來在本地搭建和運行一個簡版的 Kubernetes 叢集環境，而 kubectl 是一個用來和這個叢集環境進行互動的命令列工具。

### Minikube 安裝與配置

下面就來看一下搭建和配置的過程。Mac 系統的話安裝起來比較簡單，直接使用 Homebrew 套件管理器來安裝就可以了。首先輸入 `brew update` 來更新一下 Homebrew，然後輸入 `brew install minikube` 來安裝 Minikube。這裡由於之前已經安裝過了，所以執行得非常快。如果大家是第一次安裝的話，時間會稍微長一些。另外由於 Minikube 會依賴 kubectl，所以在安裝 Minikube 的時候會自動安裝 kubectl，不需要再單獨安裝了。

有一點需要注意的是，安裝 Minikube 有一個前提，就是你的系統必須要支援虛擬化。如果不支援的話，需要先開啟虛擬化功能或者升級系統才可以。如果不太了解虛擬化是什麼，可以先去看一下我們之前的 Docker 課程。

安裝完成之後，輸入 `minikube version` 來查看一下版本。如果能夠看到這裡顯示的版本號，就說明 Minikube 已經安裝成功了。我們再輸入一個 `minikube` 然後回車，就可以提示我們 Minikube 的一些常用命令，比如 `minikube start` 就可以啟動一個 Kubernetes 叢集，`minikube stop` 可以停止叢集，`minikube delete` 就可以刪除叢集等等，非常簡單。

先來清除一下螢幕，然後輸入 `minikube start` 來啟動一個本地的 Kubernetes 叢集。這個過程會稍微需要一點時間，因為它需要下載一些映像檔和組件，所以需要耐心等待一下。

這裡有兩個問題需要提示一下。一個是由於國內網路環境的原因，有的同學可能會遇到下載映像檔失敗的情況，這個時候可以在後面加上 `--image-mirror-country=cn` 參數來指定映像檔源為國內，就可以解決這個問題了。另外一個比較常見的問題是，Minikube 需要使用到虛擬化技術，簡單來說就是需要在本地啟動一個虛擬機。這裡使用的是 Docker Engine，如果你在啟動 Minikube 的時候提示找不到 Docker 之類的錯誤的話，可以嘗試先把 Docker 啟動起來，然後再重新啟動 Minikube 就可以了。

可以看到 Minikube 已經啟動完成了，那我們就可以使用 kubectl 這個命令列工具來和叢集進行互動了。輸入 `kubectl` 就會提示我們一些常用的命令，比如 `kubectl get nodes` 就可以查看叢集中的節點。這裡顯示了我們叢集中有一個節點，也就是剛剛啟動的這個 Minikube，還有它的狀態和已經啟動的時間。我們也可以使用 Minikube 自帶的一個 `status` 命令來查看叢集的狀態，顯示的也是一樣的資訊。

到這裡 Minikube 的安裝和配置就完成了。除了使用 Homebrew 來安裝之外，其實 Docker Desktop 也自帶了 Kubernetes，只需要在設定裡面開啟一下就可以了，這裡就不再演示了。如果你使用的是 Linux 或者 Windows 系統的話，請參考我們的筆記文件來安裝一下。文件地址在影片下方的簡介中，或者也可以在公眾號直接回覆關鍵字 K8s，也可以取得到。

## 環境搭建：Multipass + K3s 多節點叢集

上節課我們了解了如何使用 Minikube 來搭建一個本地的 Kubernetes 叢集環境，但是這個環境只是一個單節點的叢集。雖然說可以用來學習和測試一些應用，但是稍微複雜一點的場景就不太適合了，有一些需要多個節點的應用。這節課我們來看一下另外一種搭建本地 Kubernetes 叢集的方式，就是使用 **Multipass** 和 **K3s** 來搭建一個多節點的 Kubernetes 叢集環境。

### Multipass 安裝

Multipass 是一個輕量級的虛擬機管理工具，可以用來在本地快速建立和管理虛擬機。相比於 VirtualBox 或者 VMware 這樣的虛擬機管理工具，Multipass 更加輕量快速，而且它還提供了一些命令列工具來方便我們管理虛擬機。

Multipass 的安裝非常簡單，只需要在官網下載對應的安裝包然後雙擊安裝就可以了，或者你也可以使用套件管理器來安裝，如 Mac 系統的 Homebrew、Windows 的 Chocolatey，只需要執行一條 install 命令就可以了。

安裝的過程還是需要一點時間的，大家耐心等待一下。等待的過程中先來簡單說一下我們接下來要做的事情：安裝完 Multipass 之後，需要使用它來建立三個虛擬機，然後在這三個虛擬機中安裝 K3s。這三個虛擬機就是我們叢集環境中的三個節點，分別是一個 Master 節點和兩個 Worker 節點，這樣就可以在這個叢集環境中部署一些稍微複雜的應用了。

好的，安裝完成了。安裝完成之後我們可以輸入 `multipass version` 來查看一下版本，看到這裡顯示的版本號就說明 Multipass 已經安裝成功了。

### 建立虛擬機

安裝完成之後，我們就可以使用 Multipass 來建立虛擬機了。可以在命令列執行 `multipass launch` 來建立一個虛擬機。這裡我們指定了虛擬機的名稱是 k3s，同時指定了虛擬機的 CPU 是兩核、記憶體是 8G、磁碟的大小是 10G，然後回車就可以了。Multipass 會自動下載一個 Ubuntu 的映像檔並且使用這個映像檔來建立一個虛擬機，這個過程可能會稍微需要一些時間，大家耐心等待一下。

建立完成之後，我們就可以使用 `multipass list` 或者 `multipass ls` 命令來查看一下當前的虛擬機列表。可以看到我們剛剛建立的 k3s 虛擬機已經在列表中了，它的狀態是運行中，同時還顯示了它的 IP 位址和使用的映像檔版本，預設使用的是 Ubuntu 22.04 這個版本的映像檔。

關於 Multipass 的一些常用命令，我們可以通過 `multipass help` 來查看。這裡大家只需要記住幾個常用的命令就可以了，最常使用的幾個命令已經放在螢幕上了，大家可以截圖保存一下。

### 配置 SSH 登入

Multipass 建立的虛擬機預設是不允許 SSH 遠端登入的。如果想要使用 SSH 登入到虛擬機的話，就需要按照下面的步驟來修改一下 SSH 的配置。配置好之後就可以使用 SSH 登入到虛擬機，甚至可以配置免密登入，再配置一下登入命令的別名，然後就可以像我現在這樣使用一個自訂的非常簡短的命令來登入到不同的虛擬機上了。注意這一步並不是必須的，如果不配置的話其實也是 OK 的，只不過每次登入到虛擬機的時候都需要輸入 `multipass shell` 命令就可以了。如果覺得這樣也 OK 的話可以跳過這一步。

Multipass 建立的虛擬機預設是沒有密碼的。我們先來為 Ubuntu 使用者添加一個密碼。首先使用 Multipass 的 `shell` 命令來登入到虛擬機中，然後執行 `passwd` 命令來為 Ubuntu 使用者添加一個密碼。回車之後會提示我們輸入兩次密碼，輸入之後密碼就設定好了。

接下來還需要修改一下 SSH 的配置。首先找到 SSH 的配置檔案，在 `/etc/ssh/sshd_config` 檔案中，然後使用 vi 或者其他編輯器打開這個檔案。找到 `PasswordAuthentication` 這一行，將 `no` 改成 `yes`，表示允許使用密碼登入。然後再找到 `PermitRootLogin` 這一行，也修改為 `yes`，表示允許 root 使用者登入。修改完之後保存退出。

保存之後需要重啟一下 SSH 服務才生效，執行 `sudo systemctl restart sshd` 命令來重啟一下 SSH 服務。這樣我們就可以使用 SSH 來登入到虛擬機了。我們來驗證一下，輸入 `exit` 退出虛擬機之後，再輸入 `ssh ubuntu@` 後面加上虛擬機的 IP 位址，然後會提示我們輸入密碼，輸入我們剛剛設定的密碼，回車之後可以看到登入成功了。

### 配置免密登入

但是每次都需要輸入密碼的話還是比較麻煩的，所以我們可以配置一下免密登入。配置的方式也非常簡單，首先使用 `ssh-keygen` 命令在本地產生一對金鑰。這個金鑰會保存在使用者根目錄的 `.ssh` 子目錄中，其中以 `.pub` 結尾的是公鑰。我們需要把公鑰複製到虛擬機中，複製的方式是使用 `ssh-copy-id` 命令，後面加上要登入的使用者名稱和 IP 位址就可以了。這裡的話需要執行 `ssh-copy-id ubuntu@` 後面加上虛擬機的位址，就會自動把本地的公鑰複製到虛擬機中。輸入密碼之後回車，然後就可以免密登入到這台虛擬機上了。

來驗證一下，輸入 `ssh ubuntu@` 後面加上虛擬機的位址，回車之後就可以直接登入到虛擬機上了，這樣就不需要每次都輸入密碼。但是每次都需要輸入這麼長的命令也是比較麻煩的，所以我們可以為這個命令設定一個別名。比如這裡就設定了一個別名叫做 `master`，設定別名的方法是使用 `alias` 命令，然後後面加上別名和這個別名所對應的命令。這樣就可以使用這個別名來代替後面的命令，也就是可以直接使用 `master` 來登入到虛擬機上了，這樣就方便多了。

這裡有個需要注意的問題，就是 `alias` 命令設定的別名是在當前終端會話中有效，退出終端之後就會失效。如果想要別名永久生效的話，就需要把這個別名添加到 `.bashrc` 或者 `.zshrc` 的配置檔案中。比如把它添加到使用者根目錄的 `.bash_profile` 或者 `.bashrc` 檔案中，使用 zsh 的同學可以添加到 `.zshrc` 檔案中。這樣在每次打開終端的時候就會自動載入這個別名了。

## kubectl 常用命令

這節課我們來看一下 kubectl 的一些常用命令。在之前的課程中，我們分別介紹了兩種部署叢集的方式，分別是 Minikube 和 K3s。你可以使用這兩種方式中的任何一種。使用 Minikube 的話，首先執行一個 `minikube start` 命令來啟動一個 K8s 叢集，然後就可以使用 kubectl 來執行任何你想執行的操作。使用 K3s 環境的同學，首先登入到 Master 節點那台虛擬機上，然後再執行 kubectl 命令。這裡使用的是 K3s 環境，但是無論是 K3s 還是 Minikube 環境都是沒有關係的，kubectl 命令是一樣的。

### 查詢資源

kubectl 後面可以加上一些參數來執行不同的操作。比如 `kubectl get` 後面加上各種資源物件的類型，就可以查看它們的狀態。後面可以是 `nodes`、`pods`、`services` 或者 `deployments` 等等。比如輸入 `kubectl get nodes` 就可以查看叢集中的節點資訊。我們來輸入一下，可以看到目前叢集中有三個節點。如果把 `nodes` 改成 `services`，就可以查看所有的服務。可以看到返回了一個名字叫做 `kubernetes` 的服務，這是一個預設的服務，安裝好叢集環境之後就會自動建立。當然如果後面換成 `pods` 的話，就可以查看叢集中所有 Pod 的狀態了。因為現在還沒有建立任何的 Pod，所以這裡顯示是空的。

### 建立 Pod 與 Deployment

下面就來看一下如何建立一個 Pod。我們可以使用 `kubectl run` 命令來建立並運行一個 Pod。比如如果想要建立一個 Pod 用來運行 Nginx 的話，就可以輸入 `kubectl run` 後面加上名字 `nginx`，然後加上一個 `--image` 參數用來指定需要使用的映像檔，這裡我們使用的是 nginx。然後再來查看一下叢集中所有的 Pod，輸入 `kubectl get pods`。可以看到這裡多了一個 nginx，狀態是 `ContainerCreating`，表示容器正在建立中。再來執行一下，可以看到顯示的是 `Running` 正在運行，表示建立已經完成了。

除了使用 `run` 命令之外，建立各種資源物件還可以使用 `kubectl create` 命令，然後後面加上各種資源物件的類型，比如 Service 或者 Deployment 等等就可以了。我們先來加上一個 `--help` 一起來查看一下幫助。查看命令的幫助文件是一個非常好的習慣，因為幫助文件一般是最準確的官方文件，可以讓你快速了解這個命令的用法。比如這裡就列出了 `kubectl create` 支援的所有資源物件的類型，包括我們之前提到的 ConfigMap、Deployment 和 Service 等等。

仔細看一下這個列表就會發現這裡面並沒有列出 Pod。這是因為 Pod 是 Kubernetes 中的最基本的資源物件，通常情況下我們並不會直接建立一個 Pod，而是使用 Pod 的上層資源物件，比如 Deployment、ReplicaSet 等等，這些資源物件會幫助我們自動建立 Pod，所以這裡就沒有列出來。

那下面我們就來建立一個 Deployment。建立的格式是 `kubectl create` 後面加上 `deployment`，表示我們要建立的是一個 Deployment，然後加上一個名字，比如 `nginx-deployment`，然後再加上一個 `--image` 參數用來指定要使用的映像檔，這裡我們使用的是 nginx，那麼就會從 Docker Hub 上拉取最新的 Nginx 映像檔。如果大家對於映像檔還不太了解的話，推薦去看一下我們之前的 30 分鐘 Docker 入門課程，然後再回來繼續學習，效果會更好一些。

好的，那我們來執行一下，可以看到返回了這個 Deployment 建立成功了。再來查看 `kubectl get deployments`，可以看到我們的叢集中就多了一個叫做 nginx 的 Deployment。在 Ready 這一列表示顯示的是 0/1，表示這個 Pod 還沒有啟動就緒，需要再等一等，或者是啟動的過程中遇到問題失敗了。這個時候就需要我們去查看一下 Pod 的日誌來查看具體的原因了。

### Deployment、ReplicaSet 與 Pod 的關係

再來查看一下叢集中所有的 Pod，輸入 `kubectl get pods`，可以看到這裡多了一個 Pod，狀態是 Running 正在運行。當我們建立一個 Deployment 的時候，Kubernetes 會根據我們剛剛的配置來建立一個 Pod。我們只是告訴它要使用的映像檔是 nginx、名字是 nginx-deployment，剩下的一些配置都是 Kubernetes 自動幫我們建立的。

其實在 Deployment 和 Pod 之間，還有一個中間層叫做 **ReplicaSet**，它的作用是用來管理 Pod 的，在剛剛也被自動建立了。可以輸入 `kubectl get replicasets` 來查看一下，可以看到這裡也多了一個 nginx 的 ReplicaSet。這個 ReplicaSet 中有一個 Pod，也就是我們剛剛建立的那個 nginx 的 Pod。

這個 ReplicaSet 的名字是根據我們剛剛建立的 Deployment 的名字來命名的，後面加上了一個隨機的字元串用來區分不同的 ReplicaSet，它是根據 Pod 模板的 Hash 值來產生的。Pod 的名字裡面也包含了這個 Hash 值。大家仔細觀察一下這個名字就不難發現，Pod 的名字分為了三個部分：最前面的是我們剛剛建立的 Deployment 的名字，中間是 ReplicaSet 的 ID，最後是 Pod 自己的 ID。這樣從名字上我們也可以看出來這個 Pod 是屬於哪個 ReplicaSet，而這個 ReplicaSet 又是屬於哪個 Deployment 的了。

ReplicaSet 是用來管理 Pod 的副本數量的。Replicas 本身就是副本的意思。ReplicaSet 並不需要我們手動去建立，而是通過 Deployment 來完成各種配置或者管理。

再來梳理一下這幾個資源物件之間的層次關係：**Deployment** 管理的是 **ReplicaSet**，ReplicaSet 管理的是 **Pod**，而 Pod 是 Kubernetes 中最基本的資源物件，是容器的一個抽象，裡面可以有一個或者多個容器，絕大多數情況下都是一個 Pod 裡面只有一個容器。而所有 Deployment 下面這些資源物件或者組件，都應該被 Kubernetes 自動建立和管理，我們一般不會直接建立或者操作它們。

### 編輯 Deployment

接下來我們來看一下如何使用 kubectl 命令來編輯一個 Deployment。這種方式並不是很常用，大家了解一下就可以，後面會介紹更常用的使用配置檔案的方式。

比如如果我們想要修改一下剛剛建立的這個 Deployment，就可以使用 `kubectl edit` 命令，後面加上 `deployment`，表示要編輯的資源物件是一個 Deployment，然後加上 Deployment 的名字，這裡就是 `nginx-deployment`。回車之後就會使用系統預設的編輯器來打開一個 YAML 的配置檔案。老師這裡使用的是 Vim 編輯器，如果配置了其他編輯器的話也是一樣的。

這個配置檔案是 Kubernetes 自動產生的，裡面包含了我們剛剛建立的這個 Deployment 的所有配置，比如 Deployment 的名字、指定的映像檔等等。配置檔案的格式、結構以及這些配置項的具體含義，我們會在下節課介紹，這裡大家先不用太關注。我們只需要往下翻一下找到 `spec` 下面的 `replicas` 這個配置項，它是用來指定 Pod 的副本數量的，我們把它的值改成 3。修改的方式是按下 `r` 鍵切換到 Vim 的替換模式，然後輸入一個 3 就會把當前的值替換成 3，然後按一下 `ESC` 鍵退出替換模式，再輸入 `:wq` 表示保存退出。

保存之後 Kubernetes 就會自動幫我們更新這個 Deployment。可以輸入 `kubectl get pods` 來查看一下，可以看到結果中除了之前的 Pod 之外，又多出兩個新的 Pod，從最後一列的建立時間可以看出是我們剛剛建立的。然後再來看一下 ReplicaSet，輸入 `kubectl get replicasets`，可以看到裡面 Pod 的數量也都變成了三個。

從這樣的結果中也可以看出來，我們只是修改了一下 Deployment，然後 Kubernetes 就自動幫我們更新了 ReplicaSet 和 Pod。這就是 Kubernetes 一個最基本的特性：我們只需要告訴它我們想要的狀態是什麼，然後它就會自動幫我們去實現這個狀態，而不需要去關心具體的實現細節，這樣就可以大大簡化我們的工作量。

### 除錯命令

在部署環境的過程中經常會遇到各種各樣的問題，無論是開發人員還是運維人員，除錯和排查問題的能力都是非常重要的。下面就來看一下 Kubernetes 中的一些常用的除錯命令和技巧。

首先來看一下如何查看日誌。比如我們想要查看一下剛剛建立的這個 Pod 的日誌，就可以使用 `kubectl logs` 命令後面加上 Pod 的名字。回車之後就能夠看到這個 Pod 的日誌了，這裡顯示的是 Nginx 的啟動日誌。遇到問題的時候就可以查看一下具體的問題原因是什麼，大部分的時候通過日誌其實都可以得到一些有用的資訊來幫助我們定位和解決問題。

另一個非常有用的除錯命令是 `kubectl exec` 命令，它可以幫助我們進入到 Pod 的容器中，然後在容器中執行命令。比如我們想要進入剛剛建立的這個 Pod 的容器中，就可以使用 `kubectl exec` 命令後面加上一個 `-it` 參數表示使用互動式終端，再加上 Pod 的名字和一個 `-- bash` 參數。回車之後就可以看到終端的命令提示符發生了變化，表示我們已經進入到 Pod 的容器中，使用的是一個 root 使用者的 bash 終端。這個時候我們就可以在容器中執行各種命令了，比如我們可以輸入 `ls` 來查看一下當前目錄下的檔案，也可以進入到 Nginx 的配置檔案所在的目錄來查看配置檔案的內容等等。退出的時候直接輸入 `exit` 就可以了。

### 刪除資源

了解了如何建立和管理資源物件之後，我們再來看一下如何刪除資源物件。刪除資源物件也非常簡單。先來查看一下所有的資源物件，剛剛我們使用 `kubectl get` 命令後面加上各種資源物件的類型就可以查看相應的資源物件的狀態。除了各種資源物件的名稱之外，還可以加上一個 `all` 來表示查看所有內容。我們來執行一下，可以看到這裡列出了當前叢集中的所有資源物件。

現在叢集中有一個 Deployment、一個 ReplicaSet、一個 Service 和三個 Pod。那我們就來刪除一下這個 Deployment，可以使用 `kubectl delete` 命令來刪除各種資源物件。比如想要刪除剛剛建立的這個 Nginx 的 Deployment，就可以在後面加上 `deployment`，最後再加上它的名字就可以了。回車之後提示我們這個 Deployment 已經被刪除了。再來查看一下叢集中所有的 Deployment，找到我們剛剛輸入的查詢命令，回車之後就可以看到這個 Deployment 已經被刪除了，相應的 ReplicaSet 和 Pod 也都被自動刪除了。

## YAML 配置檔

上幾課我們學習了如何使用 kubectl 命令列工具來建立 Kubernetes 叢集中的各種資源物件。比如想要建立一個 Nginx 的 Deployment 的話，就需要輸入 `kubectl create deployment` 後面加上它的名字，然後再加上 `--image=nginx` 表示使用哪個映像檔。如果還有其他配置的話，可以在命令列後面繼續加上參數的內容，比如 `--replicas=3` 表示副本數量是 3 個。

這些只是最基礎的配置，而生產環境的配置一般都會比較複雜，包括了很多的配置項。如果把這些配置項都寫在命令列中的話，顯然是不太方便也是不太實用的。所以我們一般都會把這些配置寫到一個配置檔案中，然後使用這個配置檔案來建立和管理資源物件。

### 配置檔案結構

下面就來看一下配置檔案的使用方法。首先來建立一個配置檔案，配置檔案的名字並沒有什麼特別的要求可以隨便取，但是一個比較好的習慣是，檔案名字最好能夠反映出這個配置檔案的作用。比如我們想要建立一個 Nginx 的 Deployment 的話，就可以叫做 `nginx-deployment.yaml`。可以使用任何你喜歡的編輯器來編輯這個檔案，比如 Vim 或者 VS Code 都可以。為了節省時間，我們這裡就把事先準備好的一個配置檔案直接複製過來。

這是一個最基礎的配置檔案，我們來看一下它的內容：

- **apiVersion**：它是用來指定使用哪個版本的 API，也就是定義了這個配置檔案怎樣來同 API Server 進行互動。它的格式是使用斜線風格的 group 和 version，分別用來表示 API 的組別和版本。比如這裡就表示組別是 `apps`、版本是 `v1`。除了 `apps` 之外，還有很多其他的組別，比如 `batch`、`autoscaling` 等等，分別表示批次處理或者自動擴縮容。這個基本上是固定的，根據每個資源物件的類型而不同。而後面不同的版本之間可能也會有一些細微的差別，比如某一個版本中的某個配置項到了另外一個版本中可能就已經不存在了。

- **kind**：這個是用來指定資源物件的類型的。比如這裡使用的是 `Deployment`，就表示這個配置檔案是用來建立一個 Deployment。其他還有比如 Service、ConfigMap 等等各種類型，大家可以在官方文件或者我們的筆記中查看。

- **metadata**：這個是用來定義資源物件的中繼資料，比如資源物件的名稱、標籤、命名空間等等。

- **spec**：它是 specification 的縮寫，中文是規範的意思，用來定義各種資源物件的配置資訊，包括有多少個副本、使用哪個映像檔、暴露哪個 Port 等等。注意這裡的 `spec` 是一個巢狀的結構，它裡面還有一個 `spec`。第一個 `spec` 是用來定義 Deployment 的配置資訊，而第二個 `spec` 是用來定義 Pod 的配置資訊。大家可以回想一下，我們在之前的課程中提到過 Deployment 下面還有一個 ReplicaSet，這個層級關係就是通過這種巢狀的方式來體現的。Deployment 下面的 `replicas` 值是 3，表示有 3 個副本。再下面的 `template` 就定義了 Pod 的配置資訊，比如使用的映像檔是 1.25 版本的 Nginx，對外暴露的 Port 是 80 等等。

### 使用配置檔案建立資源

以上就是配置檔案的一個基本的結構。定義好這個配置檔案之後，就可以通過 `kubectl create` 命令後面加上一個 `-f` 參數來指定這個配置檔案，這樣就可以使用這個配置檔案來建立資源物件了。

我們來執行一下，執行完成之後再來看一下叢集中的資源物件，可以看到成功建立了一個 Deployment，以及它下面的 ReplicaSet 和 Pod。這就是使用配置檔案來建立資源物件的一個基本的過程。

刪除資源物件也非常簡單，只需要把 `create` 換成 `delete` 就可以了。那我們來刪除一下，這樣就可以把配置檔案中相關的資源物件都刪除掉。再來查看一下當前的狀態，可以看到剛剛建立的所有的資源物件都已經被刪除掉了。

### kubectl apply 命令

除了使用 `kubectl create` 命令之外，我們還可以使用 `kubectl apply` 命令來應用一個配置檔案中的內容。`apply` 命令會把這個配置檔案中的內容應用到叢集中，Kubernetes 同樣會根據配置檔案中的內容來建立或者更新資源物件。

還是以剛剛的配置檔案為例，我們再來使用 `apply` 命令來執行一下。執行完成之後再來看一下當前的狀態，可以看到和剛剛一樣，也建立了一個 Deployment，以及它所對應的 ReplicaSet 和 Pod，表示我們的配置檔案已經成功應用到叢集中了。

這裡需要提示的一點是，`create` 和 `apply` 命令之間還是有一些細微的區別的。`create` 命令一般只是用來建立資源物件，而 `apply` 命令則是用來根據配置檔案的內容來建立或者更新資源物件。如果資源物件不存在的話就會建立一個新的；如果資源物件已經存在但是和配置檔案中的內容不一致的話，就會自動地去更新資源物件的配置，讓它和配置檔案中的內容保持一致。

比如我們來修改一下配置檔案中的內容，打開配置檔案之後找到 `replicas` 這個選項，直接改成 5 然後保存退出。這個時候我們只需要再執行一下 `kubectl apply` 命令，修改後的內容就會自動地應用到叢集中。然後再來查看一下當前的狀態，可以看到 Pod 的數量已經變成了 5 個。

怎麼樣，是不是非常方便呢？比如在我們的生產環境中，突然遇到了一個流量的高峰，那麼這個時候就可以直接修改一下配置檔案提高副本的數量，就可以非常迅速地擴容了。這也是 Kubernetes 非常強大的一個特性。在高級一點的話，我們可以根據資源的利用率來實現自動擴縮容。

### 自愈能力

Kubernetes 是如何做到這一點的呢？其實原理也非常簡單，就是通過不停地去檢查資源的狀態，然後比較狀態和配置是否一致。如果不一致的話，就會自動地去修改資源物件的狀態，讓它和配置保持一致。這也就是 Kubernetes 的**自愈能力**。簡單來說就是 Kubernetes 可以自動地去修復自己，它會不停地去檢查和對比資源物件與配置的狀態，如果發現不一致的話就會自動地去修復，讓它們保持一致。

除了 Deployment 之外，其他的資源物件，比如 Service、ConfigMap 等等，它們的配置檔案結構也都是類似的，只是裡面的內容會根據類型的不同而稍微有一些差別。比如 Service 裡面就沒有 `replicas` 這個配置項，ConfigMap 中就沒有 `selector` 等等。大家在使用到的時候去參考一下官方文件給出的範例就可以了，這裡就不再一一介紹了。

## Service 實戰

上幾課我們通過配置檔案部署了一個 Nginx 的 Deployment。先來看一下叢集的狀態，可以看到現在有 5 個 Pod 正在運行。在之前的課程中我們了解了 Pod 是運行在節點上的，可以在查看 Pod 的時候加上一個 `-o wide` 參數來執行一下，這樣就可以看到 Pod 所在的節點以及 Pod 的 IP 位址了。

可以使用 `curl` 命令後面加上任意一個 Pod 的 IP 位址，就可以存取到這個 Pod 所提供的服務。比如我們來存取一下第一個 Pod，可以看到成功回傳了 Nginx 的預設頁面。再來存取一下第二個 Pod 的 IP 位址，也是可以正常存取的。

### 為什麼需要 Service

在之前的課程中我們也提到過，一般最好不要直接使用 Pod 來提供服務，它會存在一些問題。首先就是 Pod 使用的是一個叢集內部的 IP 位址，這個 IP 位址是不能直接從叢集外部存取到的。因為現在是在叢集內部的 Master 節點上執行的命令，所以可以直接存取到 Pod 的 IP 位址以及它們所提供的服務，但是如果換成叢集外部的話是存取不到這些服務的。

比如我們執行一個 `exit` 命令來退出到叢集外部，然後再來執行一下剛剛的 `curl` 命令，可以看到這個時候就已經存取不到了。我們來 ping 一下這個 Pod 的 IP 位址，可以看到也是 ping 不通的。這就是我們提到的，這是一個叢集內部的 IP 位址，在叢集外部是無法存取的。

還有一個問題就是 Pod 並不是一個非常穩定的實體，經常會被建立或者銷毀，這個時候它的 IP 位址也可能會發生變化。作為解決方案，可以使用 Service 來對外提供服務。

### 建立 Service（命令列方式）

下面就來看一下 Service 的使用。首先我們來建立一個 Service。建立一個 Service 通常有兩種方式，一種是使用 kubectl 命令列，另一種就是通過配置檔案。大部分情況下我們都是通過配置檔案來建立的，命令列的方式一般來說不是很常用，大家了解一下就可以。

那我們先來看一下命令列的方式。可以通過 `kubectl create` 來建立一個服務，後面加上 `service` 表示要建立的資源類型，然後是服務的名字以及一些參數。或者也可以使用 `kubectl expose` 命令來將一個已經存在的 Deployment 對外公開為一個服務。因為我們這裡已經有了一個 Deployment，那我們就使用 `expose` 的方式來建立一下。回車之後就可以看到服務已經建立成功了。

我們再來看一下叢集的狀態，可以看到這裡多了一個服務。使用 `curl` 命令來存取一下這個服務，可以看到也是沒有問題的。

如果想要查看這個服務的更詳細的資訊，可以輸入 `kubectl describe service` 後面加上它的名字。回車之後就可以看到這個服務的詳細資訊了，包括它的命名空間、類型、IP 位址、Port 等等。

除了 Service 以外，很多其他資源物件也都可以使用 `describe` 這個命令來查看。比如想要查看 Deployment 的話，就只需要把剛剛命令中的資源類型換成 `deployment` 就可以了。來執行一下，也可以看到它的詳細資訊。Pod 也是一樣的，這裡就不演示了，大家可以自己嘗試一下。

刪除一個服務也非常簡單，只需要使用 `kubectl delete` 命令後面加上 `service` 表示要刪除的資源類型，然後再加上服務的名字就可以了。來執行一下，然後再來查看一下叢集的狀態，可以看到這個服務已經被刪除了。

### 建立 Service（配置檔案方式）

接下來我們再來看一下如何通過配置檔案的方式來建立一個服務。首先需要編輯配置檔案：

- 第一行是一個固定的 API 版本。
- 然後是資源的類型 `kind`，這裡就是一個服務 `Service`。
- 下面是中繼資料 `metadata`，裡面的 `name` 就是服務的名字，這裡我們指定為 `nginx-service`。
- 然後是 `spec`，也就是服務的規格和描述部分。

首先是一個 **selector**（選擇器），它是用來選擇特定資源的。這裡我們指定的是所有 `app` 是 `nginx` 的資源。稍微解釋一下這個選擇器的作用，就是用來選擇特定的資源的，它一般和 **label**（標籤）一起使用。在剛剛 Deployment 中我們可以找到這個標籤。簡單來說就是通過這個標籤來選擇特定的資源。這裡我們指定的標籤就是 `app=nginx`，也就是選擇標籤中 `app` 的值是 `nginx` 的資源。

標籤和選擇器簡單來說就像是在超市裡面買東西一樣，我們會通過商品的標籤來選擇我們想要購買的商品，只不過這裡是通過標籤來選擇相應的資源。標籤和選擇器的概念在 Kubernetes 中非常重要，我們就是通過它們來為叢集中的資源進行分類和選擇的，大家一定要理解清楚。

然後是對外公開的 Port。這裡我們指定協議是 TCP，Port 是 80，然後是一個目標 Port（`targetPort`）也是 80。這裡雖然都是 80，但是前面這個是服務對外公開的 Port，而後面這個 `targetPort` 是這個服務後面所對應的 Pod 的 Port。

這樣就建立好了一個服務的配置檔案。保存退出之後再來應用一下這個配置檔案，然後再來看一下叢集的狀態，可以看到服務已經建立成功了。來存取一下這個服務，可以看到也是沒有問題的。

### NodePort 類型的服務

到這裡我們就使用配置檔案完成了一個服務的建立。但是這個服務還是一個內部服務，只能在叢集內部存取，在叢集外部，比如我們使用本地的瀏覽器來存取的話還是不行的。這個時候我們就需要使用另一種類型的服務，也就是 **NodePort** 類型的服務。它可以將服務公開到叢集的節點上，然後我們就可以通過節點的 IP 位址和 Port 來存取到這個服務了。

再來修改配置檔案，把這個服務改成 NodePort 的類型。打開配置檔案之後，在 `spec` 下面加上一個服務的類型，也就是 `type`。在之前我們並沒有指定這個服務的類型，那麼它預設就是一個 `ClusterIP` 類型的服務，也就是一個叢集內部的服務。那現在我們通過這個 `type` 來指定一下它的類型，把它改為 `NodePort`，也就是節點 Port 類型的服務。然後在配置檔案的最後一行再加上一個 `nodePort` 來配置節點對外提供服務的 Port，指定為 30080。注意這裡有個要求，就是節點的 Port 範圍必須在 30000 到 32767 之間，否則的話會提示我們 Port 範圍不正確。

那我們來保存一下，然後再來應用一下這個配置檔案，然後再來看一下服務的狀態，可以看到這個服務已經變成了 NodePort 類型的服務。那我們打開一個瀏覽器來存取一下，先來查看一下節點的 IP 位址，然後複製一下，輸入節點的 IP 位址和 Port 號，然後回車。可以看到這個時候我們就可以從叢集的外部存取到 Nginx 的服務了。

### 其他 Service 類型

除了 NodePort 類型的服務之外，Service 還有幾個其他的類型，包括 **LoadBalancer**、**ExternalName**、**Headless** 等等。

- **LoadBalancer** 類型的服務可以將服務公開到外部的負載均衡器上。
- **ExternalName** 可以通過返回一個 CNAME 記錄來將服務映射到一個外部域名上。
- **Headless** 可以建立一個沒有 ClusterIP 的服務，主要用於 DNS 解析和服務發現。

時間關係這裡就不一一詳細介紹了。

## Namespace 命名空間

下面我們來做一個小案例，使用 Kubernetes 來安裝一個可視化的 Kubernetes 管理工具，用自己來管理自己。工具的名字叫做 **Portainer**，安裝的方法也非常簡單，只需要在官網下載一個配置檔案，然後直接應用一下就可以了。這裡我們已經把配置檔案提前下載好了，在課程資料中也可以找到。那我們就直接執行一下。

注意這裡有個 `-n` 參數，它是用來指定命名空間的。稍微解釋一下**命名空間（Namespace）**，它是用來對叢集中的資源進行隔離和分組的一種機制，這樣在不同命名空間中的資源就不會因為名字相同而產生衝突了。一般可以用來隔離不同的專案或者不同的環境。

查看命名空間的命令是 `kubectl get namespaces`，也可以簡寫成 `kubectl get ns`。那我們來執行一下，可以看到當前叢集中有五個命名空間。除了剛剛建立的 `portainer` 命名空間之外，其餘四個命名空間都是 Kubernetes 叢集預設建立的。所有資源物件如果不指定命名空間的話，就會被放到 `default` 這個預設的命名空間中。我們平常輸入的各種命令，如果不指定命名空間的話，預設也都是在 `default` 命名空間中執行的。

比如我們在查看叢集狀態的時候，因為沒有指定命名空間，所以預設是在 `default` 命名空間中查看的。而我們剛剛安裝的 Portainer 在配置檔案中指定了命名空間為 `portainer`，所以我們在查看叢集狀態的時候也需要加上一個 `-n` 來指定要查看的命名空間。可以看到我們剛剛建立的都已經啟動了，包括一個 Pod、一個 Service、一個 Deployment 和一個 ReplicaSet。這樣安裝就完成了。

我們使用瀏覽器來存取一下。打開瀏覽器，輸入 Master 節點的 IP 位址和一個 30777 的 Port 號，然後回車，就可以看到我們成功存取到了 Portainer 的管理介面。如果是第一次存取的話會要求我們設定一個密碼，長度至少 12 位。我們來設定一下，輸入密碼之後點擊 Create User 建立使用者按鈕，然後就成功登入進來了。

點擊一下左邊的按鈕就會連線到我們本地的叢集環境，右邊的新增按鈕可以幫助我們連線到其他的叢集環境，比如如果你還有部署在 Google 或者 AWS 上的叢集的話就可以使用它來配置一下。那我們選擇左邊的快速開始，然後就會進入到我們的叢集管理介面。點擊一下這個環境列表中的 `local` 這一行就可以進入到一個叢集的概覽頁面。在這裡可以看到我們當前這個叢集的一些概覽資訊，包括它的命名空間、應用數量、節點數量等等。

在左側導航選單中也提供了很多功能，包括命名空間、服務、Ingress、ConfigMap、Secret 等等。這些都是我們在前面的課程中提到過的。有幾個大家沒有接觸過的功能，比如 **Helm**，它是一個 Kubernetes 的套件管理工具，可以幫助我們更方便地管理 Kubernetes 的應用，有點類似於 Linux 系統中的 apt 或者 yum，可以幫助我們更加方便地安裝、升級和管理應用。基本上常用的 K8s 應用都可以在 Helm 中找到。

我們當前叢集的一些資訊，包括當前的節點數量、CPU 和記憶體的使用情況、節點的 IP 位址以及當前的狀態等等。下面還有一些設定功能，用來設定我們的使用者、團隊、角色，以及管理環境或者倉庫等等。這裡就不一一介紹了。

然後建議大家根據自己的需求和實際情況來深入學習和使用它的某一個方面。畢竟每個人的需求和實際情況都是不太一樣的，有的人是架構師，可能需要全面系統地了解它的方方面面；而有的同學可能只是想學習一下 Kubernetes 的基本概念，了解一下當下最流行的容器編排技術。

我們的建議是先有一個整體的了解，然後再根據自己的實際需求來深入學習你感興趣或者實際工作中需要使用到的那部分內容，這樣會更加高效和有針對性。

這期內容就到這裡，我們下期再見。
